# 付録B — ソースレジスタ（出典一覧）

本書の主張は外部の出典に依拠しており、誠実であるためには、どの出典かを明示する必要がある。このレジスタには、本書が依拠する資料を、それぞれが何をもたらすかの一行注記とともに列挙する。あわせて、決して混同してはならない2種類の出典を区別する。**一次出典**は権威があり永続的なものである。標準化団体、政府機関、そして規範的なガイダンスや仕様を公開するプラットフォームベンダーがこれにあたる。本書の SHALL が立脚する地盤である。**二次シグナル**は時宜性はあるが暫定的なものである。脆弱性フィード、ベンダーの脅威リサーチ、業界分析がこれにあたる。いま何が起きているかを伝え、ランタイムと更新のループに情報を供給するが、それ単体では規範的な主張の根拠にはならない。

この区別は学術的なものではなく、運用上のものである。本書が SHALL と言うとき、その背後には一次出典がある。本書が動く標的（MCP認可モデル、EMA標準、今月の攻撃手法）を記述するとき、その背後には二次シグナルがあり、本書はその旨を明記する。両者が食い違う場合は、要件については一次出典が、ランタイムポスチャについては二次シグナルが基準となる。

## 一次出典

これらは、要件仕様の土台となる権威ある参照資料である。第9章で要件に出典マッピングが付されている場合、それはここを指す。

**NIST** — AIリスクマネジメントフレームワークとその生成AIプロファイル（AI 600-1）は、ガバナンスの章が用いる govern/map/measure/manage の構造とリスクマネジメントの語彙を提供する。NIST の CAISI および NCCoE による、AIとソフトウェアエージェントのアイデンティティに関する成果は、アイデンティティと委譲の要件を支えている。*CT-01, CT-03, CT-05, CT-13, CT-15 およびガバナンス系要件の基盤。*

**CISA and NCSC** — AIシステムを安全にデプロイするための共同および単独のガイダンス、ならびにセキュア・バイ・デザイン原則。最小権限、ロギング、インシデント対応の各要件の主たる根拠であり、「モデルではなくシステムを守る」という多層防御の立場の根拠でもある。*CT-05, CT-09, CT-12 の基盤。*

**OWASP GenAI / LLM Top 10** — LLM01 のプロンプトインジェクション項目と、より広範な GenAI セキュリティプロジェクトは、インジェクション、RAGポイズニング、ツール濫用という脅威クラスの標準的な語彙を与える。*TH-01, TH-02, TH-03, TH-04 および CT-07 の基盤。*

**MITRE ATLAS** — AIシステムに対する敵対的戦術のナレッジベース。攻撃者の手法によって脅威を枠付けるための、また評価回避クラスのための参照資料。*脅威分類と CT-11 の基盤。*

**CSA (Cloud Security Alliance) — MAESTRO** — エージェント型AIの脅威モデリングフレームワーク。マルチエージェント要件の背後にある、委譲と信頼継承の分析の参照資料。*TH-06 と委譲縮小ルールの基盤。*

**Microsoft** — 安全なエージェントシステムに関するガイダンスと、エージェント固有のアイデンティティに関する Entra Agent ID の取り組み。ワークロードアイデンティティ、ツールレジストリ、多層コントロールの参照資料。*CT-02, CT-03, CT-04, CT-08 の基盤。*

**Anthropic** — エージェントとツールの安全性、サンドボックス化、コンテンツ境界の扱いに関する公開ガイダンス。実行の分離と入出力境界の参照資料。*CT-06, CT-07 の基盤。*

**MCP specification and the EMA standard** — Model Context Protocol 仕様（2025年の認可関連改訂を含む。すなわち PKCE 付き OAuth 2.1、リソースインジケータ、トークンパススルーの禁止）と、認可をアイデンティティプロバイダへ引き戻すエンタープライズ認可標準である Enterprise-Managed Authorization。これらは一次出典だが、*動いている*。登場が新しく、まだ固まりきっていないため、本書は構造については規範として引用しつつ、詳細は変わりうると明記している。*CT-14 と MCP 要件（REQ-050, REQ-014）の基盤。*

## 二次シグナル

これらは、モニタリング（REQ-032）、継続的更新プロセス、ランタイムポスチャに情報を供給する、時宜性はあるが暫定的な出典である。現在の状況の証拠として引用するものであり、SHALL の唯一の根拠として引用することは決してない。

**脆弱性・悪用フィード** — 公開済み脆弱性の CVE、実際に悪用されているものを示す CISA の KEV カタログ、悪用確率の EPSS。ランタイムポスチャの `critical_vulnerability` および `active_exploitation` トリガーが作動する際の入力である。*CT-10、REQ-032、第8章のトリガーに供給。*

**ベンダーおよび研究機関の脅威リサーチ** — Palo Alto Unit 42、CrowdStrike、Wiz、Lakera、Trail of Bits といったグループによるセキュリティリサーチ。エージェント固有の攻撃手法（ツールポイズニング、ラグプル、インジェクションの亜種）を、出現の都度報告する。最新性の点で価値があるが、確定した事実ではなく、検証すべきシグナルとして扱う。*CT-10、CT-11、評価スイートに供給。*

**MCPエコシステムの計測** — デプロイされた MCP サーバの認可ポスチャを報告する調査（たとえば、大きな割合のサーバが認証をまったく備えていない、あるいは長寿命の静的キーに依存している、といった報告）。第5章で MCP 要件の動機付けに用いる。変化が見込まれるエコシステム状態のスナップショットとして引用する。*MCP 設定チェックリストに供給。*

**業界分析** — Deloitte、McKinsey、Accenture といった企業による導入とリスクの分析。規模と趨勢を捉えるうえで有用だが、技術要件の出典ではない。*文脈情報のみ。*

## カバレッジとギャップに関する注記

出典の基盤が最も強固なのは、複数の一次出典が収斂するアイデンティティ、ツールの安全性、インジェクション系統の領域である。一方で、読者が知っておくべき2つの面で手薄である。第一に、ランタイムポスチャの設計（第8章）は、NIST と CISA の一般的なリスクマネジメントの姿勢を特定のステートマシンへと総合したものであり、直接の引用ではなく本書独自の構成である。それらの一次出典に根ざしてはいるが、その先まで踏み込んでいる。第二に、MCP と EMA に関する記述は、現行かつ権威あるが、なお発展途上の標準に依拠している。したがって、そこで引用される個別のメカニズムは、コンプライアンス上の主張の根拠として頼る前に、最新の仕様と照合し直すべきである。どちらのギャップも、取り繕うのではなくここに明示している。確信がある場合にのみ SHALL を使い、確信がない場合はそう明言するという本書のルールに沿うものである。出典の帰属に誤りが判明した場合の正誤表プロセスは、変更履歴が担う。
