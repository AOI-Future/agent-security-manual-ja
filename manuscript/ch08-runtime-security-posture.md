# ランタイムセキュリティポスチャ

## 知ることと直すことの間の空白

あらゆるセキュリティプログラムには、自分たちが危険にさらされていると知った瞬間から、レビューとテストを経た修正が本番に反映される瞬間までの間に、空白の期間が存在する。従来のソフトウェアであれば、この期間は不快なものにとどまる。エージェントにとっては危険である。エージェントはその間もずっと自律的に行動し続ける。信頼できないコンテンツを読み、ツールを呼び出し、金銭を動かす。しかもその根拠は、いまや誰もが時代遅れだと知っている仕様なのだ。仕様が正式に改訂されるのを待つことは、もはや信頼していないルールの下で、特権を持つ自動化アカウントを走らせ続けることを意味する。

本章は、この空白を第二のコントロールループで閉じることを扱う。本書の通常のループは遅く、慎重である。脅威を分析し、コントロールを設計し、要件を書き、テストを追加し、変更をレビューして出荷する。これは正しいプロセスであり、急ぐべきではない。急げば粗悪なコントロールを出荷することになるからだ。しかし、進行中の攻撃への唯一の対応手段とするには遅すぎる。そこで、これと並行して第二のループを走らせる。**証拠に基づいてエージェントの能力を一時的に低下させる高速なランタイムループを、遅いループが恒久的な答えを出すより前に作動させるのである。** ランタイムポスチャがその高速ループである。遅いループが本来の仕事をきちんと果たすための時間を稼ぐのだ。

## 二重ループ設計

2つのループを別物として保つことが、中心的な設計判断である。両者の故障モードは正反対だからだ。遅いループ（仕様）は慎重で、レビューを経て、恒久的でなければならない。その危険は、速く動きすぎて悪いルールを成文化してしまうことにある。速いループ（ランタイム保護）は即時かつ可逆でなければならない。その危険は、動きが遅すぎてエージェントを無防備なまま放置してしまうことにある。両者を一体化させてしまえば、つまりランタイムでのあらゆる反応が恒久的な要件になってしまえば、遅いループは変更の乱発で麻痺するか、レビューされていないルールで汚染されるかのどちらかである。したがってランタイムループは明示的に暫定的である。そこで行われることはすべて一時的であり、そのすべてが記録され、後で遅いループが裁定する。

だからこそランタイムポスチャは、再構成ではなく能力の*低下*なのである。エージェントのポリシーを書き換えるのではなく、能力をしばらくの間絞るか停止する。遅いループが追いついた時点で、その低下は本物の要件へ昇格されるか、誤報として緩和されるか、既存のルールに統合される。ただしその裁定は遅いループの中で、遅いループ自身のタイムラインに沿って行われるのであって、混乱の渦中で行われるのではない。

## ポスチャステートマシン

エージェントは少数の名前付きポスチャのいずれか1つに位置し、証拠に基づいてポスチャ間を移動する。各状態は、全能力から全停止に至るはしごを成す。

**normal**（通常）— 現行仕様の下での全能力。デフォルトの状態である。

**watch**（監視）— 何も制限されないが、モニタリングが強化され、シグナルが収集される。緊急性の低いインテリジェンスが届く場所である。ベストプラクティスの更新、ドキュメントの変更、注視には値するがまだ行動には値しない弱いアノマリーなどだ。

**provisional_guard**（暫定ガード）— レビュー待ちの間、選択された能力が引き締められる。信憑性はあるが未確認の懸念に対するポスチャである。高リスクのガイダンスが変わった、あるいはアノマリーが本物に見えるがまだ攻撃と確定していない、といった場合だ。エージェントは特定の権限を制約されたまま働き続ける。

**restricted**（制限）— 危険な能力が大幅に切り詰められる。依存関係に確認済みのクリティカルな脆弱性が見つかった場合や、アノマリーが持続する場合に、エージェントが置かれるポスチャである。安全な仕事は引き続きできるが、高リスクのツール・送信・特権は引き上げられる。

**emergency_block**（緊急遮断）— 危険な操作面が完全に停止される。進行中の悪用に対するポスチャである。攻撃がいままさに起きており、人間が引き継ぐ間、エージェントが危害を及ぼす能力を遮断する。

**revised_baseline**（改訂済みベースライン）— 一時的なポスチャが遅いループによって裁定され、新しい通常状態が発効している。ループを閉じる出口である。ランタイムでの反応がレビューを経て仕様の一部となり、エージェントは更新されたルールの下で完全かつ正しい運用に戻る。

はしごを上る移動は速く、証拠駆動である。normal へ戻る移動は遅いループのレビューを通じてのみ起こる。攻撃を受けていたという事実が、いつの間にか忘れ去られることは決してないのだ。

## トリガークラスとその SLA

ポスチャは雰囲気では変わらない。分類されたトリガーによって変わり、各トリガーには目標ポスチャと応答時間の SLA が定められている。トリガークラスに前もって名前を付けておくことこそが反応を速くする。判断はすでに下されているからだ。オンコールの対応者は計画を実行するのであって、プレッシャーの中で計画を発明するのではない。

**active_exploitation**（進行中の悪用）— 攻撃が進行中であると観測された。目標ポスチャは emergency_block、SLA は即時、おおむね15分。まず停止し、調査は後回しにする、という順序を取る唯一のトリガーである。

**critical_vulnerability**（クリティカルな脆弱性）— モデル、フレームワーク、ツール、依存関係における確認済みのクリティカルな欠陥。特に既知悪用リストに載っているもの。目標ポスチャは restricted、SLA はおおむね1時間。

**high_risk_guidance_change**（高リスクのガイダンス変更）— 権威あるガイダンスが、リスクを実質的に変えるかたちで変化した（標準化団体が何かを再分類した、ベンダーが設計上の弱点を開示した、など）。目標ポスチャは provisional_guard、SLA はおおむね24時間。

**suspicious_runtime_anomaly**（疑わしいランタイムアノマリー）— モニタリングが、攻撃のように見えるがまだ確認されていない挙動を検出した。目標ポスチャは深刻度に応じて provisional_guard または restricted、SLA はおおむね1時間。

**best_practice_update**（ベストプラクティスの更新）— ポスチャを改善するが、能動的なリスクの兆候ではない新しいガイダンス。目標ポスチャは watch。遅いループの通常のサイクルに統合される。

**documentation_only**（ドキュメントのみ）— 能力に影響しない情報提供的な変更。目標ポスチャは watch、月次サイクルでレビューされる。

SLA は意図的に非対称である。進行中の悪用には分単位、ガイダンス変更には1日、ドキュメントには1か月。緊急度を証拠に釣り合わせることが、速いループが本物の攻撃に過小反応することも、ブログ記事1本に過剰反応することも防ぐのである。

## キルスイッチときめ細かな取り消し

はしごの頂点にはキルスイッチがある。その最も重要な性質は、単一のスイッチではないということだ。「すべて停止」という二値の道具は、往々にして間違った道具である。狭い脅威を全面的な障害に変換してしまうからであり、その障害自体、攻撃者が課そうとしているコストかもしれない。ランタイムポスチャは代わりに*きめ細かな*取り消し（granular revocation）を提供する。個々の能力——特定のツール、MCP サーバー、外部送信、メモリ書き込み、RAG 検索、委譲、スケジュール実行や cron 実行、あるいはエージェントユニット全体——を、それぞれ独立に遮断する能力である。

きめ細かさこそが、対応を脅威に適合させる。ある MCP サーバーでのラグプル（rug-pull）には、そのサーバーだけを取り消し、他には手を付けない。メモリ汚染イベントでは、読み取りは継続させたままメモリ書き込みを凍結する。持ち出し（exfiltration）の試みには、エージェントが安全な内部作業を続けたまま外部送信を遮断する。エージェント全体を止めるのは、本当にシステム全体に及ぶ侵害の場合だけである。これらはいずれも、同じキルスイッチ機構を異なる能力に向けたものであり、それぞれが、第4章のスコープ付きクレデンシャル設計によってそもそも取り消し可能になっていた、きめ細かなクレデンシャルの1つに対応する。アイデンティティ設計が運用面で報われるのはここである。狭く付与していたからこそ、狭く取り消せるのだ。

## ポリシーバージョンの紐付け、シグナルキュー、リコンシリエーション

3つの機構が、速いループを誠実に保ち、遅いループとつなぎ続ける。

**ポリシーバージョンの紐付け（policy version binding）**とは、すべてのランタイムアクションが、その実行時にどのバージョンの仕様とポスチャポリシーが有効であったかを記録することを意味する。これがなければ、ポスチャの変更によって監査証跡は曖昧になる。あるアクションがその時点で許可されていたのかどうかを判別できないのだ。これがあれば、記録されたすべてのアクションを、それを支配していたまさにそのルールに照らして解釈できる。監査人とインシデントレビュアーの双方が必要とするのはこれである。

**ランタイムシグナルキュー**は追記専用の記録——実務上は `runtime_signal_queue.jsonl` ——であり、すべてのトリガー、ポスチャ変更、能力低下が、発生と同時にそこへ書き込まれる。速いループの記憶であり、遅いループの受信箱である。ランタイムが行うことに不可視のものはなく、反応からレビューまでの間に失われるものもない。

**リコンシリエーション（reconciliation）**は、遅いループがキューを順に処理し、各エントリに帰結を割り当てる処分のステップである。恒久的な要件へ昇格する、恒久的な条件付きルールとして成文化する、誤検知としてマークして低下を緩和する、既存の要件に統合する、あるいはインシデントのフォローアップとして持ち越す。リコンシリエーションこそが、エージェントを一時的なポスチャから `revised_baseline` へと移すものであり、暫定ガードが一度もレビューされないまま、いつの間にか恒久化してしまうことを防ぐ規律である。

これらすべてを束ねる設計上の禁止事項が1つある。**ランタイムループは能力を減らすことしかできず、拡大することは決してできない。** 新しい権限を付与できる速いループは、レビューを経ない特権昇格経路になってしまう。それこそ、本書全体が防ぐために組み立てられているものにほかならない。ランタイムポスチャはエージェントを絞ることも止めることもできるが、上げることは決してできない。上げるのは遅いループの仕事であり、レビューを伴い、その終着点は常に `revised_baseline` だけである。

ランタイムポスチャは、第1章の5番目の問い——修正が追いつかない速さで攻撃されたとき、我々はどうするのか——への運用上の答えである。これがあれば、答えは全員招集のパニックではなく、名前付きのトリガーと SLA を備えた、定義済みのステートマシンになる。これで運用の全体像は完成である。第IV部は、本書全体を監査可能なものへと変えていく。その出発点が要件仕様、すなわち規範的な SHALL（しなければならない）の集合である。実際に実行していることを誰かが検証できるようにするためには、第II部と第III部のすべてのコントロールがこの SHALL として表現可能でなければならないのだ。
