# 検証とテスト

## SHALL を合格/不合格に変える

テストできない要件は、通し番号の付いた意見にすぎない。第9章はすべての SHALL が検証可能であるように書かれた。本章は、その検証が実際に息づく場所である。その役割は、各要件に対して、曖昧さのない合格/不合格 (pass/fail) を返すテストを少なくとも1つ結びつけることである。これによって「我々は準拠している」は意図の表明であることをやめ、あなたを信頼しない相手に示せる結果になる。これは本書が第1章以来課してきた連鎖、すなわち「脅威 → コントロール → 要件 → 検証」の最後の環であり、他の3つを誠実なものにする環である。

この計画は、評価に嘘をつかせようと狙い定めてくる敵対者 (TH-09) との接触に耐えるように構築されている。すなわち、本番環境を代表する条件に対して実行されるテスト、一回限りではなく反復可能なテスト、そしてそれ自体が改ざんされて偽の合格になることのない結果の記録である。本章では4つの検証手法を説明し、要件をテストにマッピングし、運用ルールを定め、その上で、全体を実行して署名付き成果物を生成するコンパニオンキットへと接続する。

## 4つの検証手法

要件はそれぞれ異なる形で破られるため、異なる種類のテストが必要になる。この計画では4つの手法を用いる。各手法には2文字のコードがあり、テスト識別子の接頭辞となる。

**VT-S — 静的・構造的検証。** 攻撃を実行することなく、設定、ポリシー、インベントリ、コードを検査する。許可リスト（allowlist）は存在するか、ツール定義はハッシュ化されているか、ポスチャのステートマシンは配線されているか、アイデンティティは人間のものと区別されているか。VT-S は「コントロールは要求どおりに構築・設定されているか」に答える。安価で決定的であり、システムの構造の性質として表れる要件に適した手法である。

**VT-D — 動的・敵対的検証。** 実際の攻撃を実行して挙動を観察する。取得コンテンツを介してプロンプトインジェクションを撃ち込み、ツールの悪用を試み、委譲を通じた権限昇格を試み、持ち出し（exfiltration）を試みる。VT-D は「誰かが押し込んできたときにコントロールは持ちこたえるか」に答える。インジェクション系とツール悪用系のための手法であり、レッドチーム用ツールが使われる場所である。

**VT-E — 証拠・ログ検証。** システムが生成する記録を検査する。エピソードをログから再構築できるか、シークレットは秘匿化されているか、相関IDは一貫して貫通しているか、すべてのランタイムアクションはポリシーバージョンに紐づいているか。VT-E は「システムは記録すべきものを、調査担当者が使える形で記録したか」に答える。あらゆるインシデント対応が依存する監査レイヤーを検証する。

**VT-A — アテステーション・監査検証。** より高次の性質を確認する。ランタイムポスチャがトリガーに応じて実際に SLA 内で遷移したか、リコンシリエーション（突合）が行われたか、ガバナンス記録が存在するか、検証スイート自体が実行され保持されたか。VT-A は「保証プロセス自体が設計どおりに機能しているか」に答える。ランタイム、ガバナンス、そしてメタ要件のための手法である。

ツールベースラインはこれらの手法を支えるものであって、定義するものではない。読み取り専用のポスチャスキャナ（**Hardshell v2**）が、自身は書き込み能力を持たないまま静的な VT-S スイープを実行する。これが重要なのは、本番への書き込みアクセスを持つ検証ツールは、それ自体が新たな攻撃面になるからである。敵対的テストは、VT-D 攻撃ライブラリとして **PyRIT**（Microsoft のリスク特定ツールキット）と **garak**（NVIDIA の LLM 脆弱性スキャナ、v0.15.1）を活用する。ツールは道具にすぎず、手法こそが契約である。

## 要件からテストへのマッピング

すべての要件は1つ以上のテストにマッピングされ、テストの接頭辞がその確認方法を教えてくれる。以下の表はそのマッピングの*代表的な*一部であり、パターンを読み取れるだけの行数であって、網羅的なマトリクスではない。すべての SHALL がテストの完全な集合に結びつけられた要件-テストのマスターマトリクス全体は、生きた成果物である。リグレッションケースが追加されるたびに成長するため、印刷されたページに凍結されるのではなく、コンパニオンキットの中に置かれている。キットはこれらの安定した識別子に対して結果を報告するとともに、カバレッジチェック (REQ-054) を通じて、テストを持たない SHALL がひとつも残らないことを強制する。

| 要件 | テスト | 手法 |
|-------------|-------|-----------|
| REQ-001 アイデンティティの分離 | VT-S-001, VT-A-001, VT-A-002 | 静的、アテステーション |
| REQ-010 ツール許可リスト | VT-S-010 | 静的 |
| REQ-011 スコープ / 入力バリデーション | VT-D-010, VT-D-011 | 動的 |
| REQ-020 信頼できないコンテンツのラベリング | VT-S-020, VT-D-020, VT-D-021, VT-D-022, VT-D-023, VT-D-024 | 静的、動的 |
| REQ-021 TTL / 鮮度 | VT-S-021 | 静的 |
| REQ-030 エンドツーエンドのロギング | VT-S-030, VT-D-030, VT-D-031 | 静的、動的 |
| REQ-040 ポスチャのステートマシン | VT-A-040, VT-A-041 | アテステーション |
| REQ-041 トリガー / SLA | VT-D-040, VT-D-041, VT-E-040, VT-E-041 | 動的、証拠 |
| REQ-050 MCP の審査 / ピン留め | VT-D-050, VT-S-050, VT-E-050, VT-E-051 | 動的、静的、証拠 |
| REQ-051 委譲の絞り込み | VT-A-050, VT-A-051 | アテステーション |
| REQ-053 承認ゲートのメトリクス | VT-A-060, VT-A-061 | アテステーション |
| REQ-054 検証カバレッジ | VT-A-070, VT-A-071 | アテステーション |

行を暗記するのではなく、パターンを読み取ってほしい。構造的な要件（許可リスト、ラベリングの存在、TTL 設定）には VT-S が付く。挙動に関する要件（ラベリングは実際にインジェクションを止めるか、バリデーションは実際に不正な引数を拒否するか）には VT-D が付き、しばしば複数になる。攻撃1つでは証明としては弱く、攻撃のファミリーがあってはじめて、コントロールが汎化することを証明できるからである。記録の完全性に関する要件には VT-E が付く。そしてプロセスレベルの要件、すなわちポスチャ遷移、委譲の絞り込み、承認メトリクス、そしてカバレッジそのものには VT-A が付く。これらの要件が主張しているのは*プロセス*が正しく実行されたということであり、それはプロセスの出力に対するアテステーションによって確認するものだからである。インジェクション系が最も多くの動的テスト（VT-D-020 から VT-D-024）を抱えているのは、まさにそれが最も塞ぎにくく、最も後退しやすいからである。ここに示していない行、すなわちメモリの可逆性、ロールバック演習、ランタイムポスチャのアテステーションも同じルールに従う。それぞれがキットのマスターマトリクスにおける動的、証拠、またはアテステーションのテストであり、その中のどの SHALL であれ未テストのまま残っていれば、カバレッジチェックは合格を拒否する。

## 運用ルール

テストが信頼に足るのは、規律のもとで実行され記録される場合に限られる。それを律するルールが4つある。

結果は `reports/<date>_verification_report.md` という日付付きレポートに書き出される。これにより、すべての実行は、書き換え可能なダッシュボードではなく、それぞれ独立した保持される成果物になる。レポートには各テストを識別子ごとに合格または不合格としてのみ記録し、それ以上は何も書かない。ペイロードも、トランスクリプトも、キャプチャしたデータも記録しない。実行した攻撃を引用する検証レポートは、その攻撃のマニュアルと化し、シークレットが漏れる場所になるからである。あらゆる不合格は `data/change_queue.jsonl` に書き込まれる。これは継続的更新プロセスとランタイムリコンシリエーションプロセスが消費するのと同じキューであり、したがって失敗したテストは、誰かが覚えていることを祈るだけの一行ではなく、自動的に追跡される作業になる。そしてシークレットは、いかなるテスト出力にも決して記録されない。テストがクレデンシャルを扱わなければならない場合、それは `[REDACTED]` とロケータの形で現れる。これは本書の他のあらゆる場所で適用されるルールと一貫している。これらのルールの要点は、検証プロセス自体が、それが検証する脅威に対して堅牢化されているということである。漏らさず、省略によって嘘をつかず、その失敗が蒸発することもない。

## ゲートではなくリグレッション

運用上もっとも重要な性質は、これがローンチ時のチェックリストではなく常設のスイートである、という点である。第7章では、モデル、ツール、RAG ソース、プロンプト、権限のあらゆる変更が、閉じたはずのケースを再び開きうるため、評価は継続的に実行されなければならないと論じた。検証計画は、その議論を実行可能にする機構である。かつて修正されたすべての失敗は、恒久的なリグレッションケースとしてスイートに残り続ける。したがって修正が静かに腐ることはなく、変更が、赤いテストという証跡を残さずに古い穴をこっそり開け直すこともできない。**REQ-054** は計画そのものに対してループを閉じる。カバレッジチェック（VT-A-070、VT-A-071）は、すべての SHALL が依然として少なくとも1つのテストにマッピングされていることを検証するため、テストを追加せずに要件だけを追加してスイートを空洞化させることはできない。それ自体が検証されていない検証計画は、単なる未確認の主張にすぎない。

## コンパニオンキットと署名付き成果物

本章の内容はすべて手作業で実施できるし、本書はそれができるように、何を確認すべきかを正確に示している。本書がやらないこと（書籍にはできないことだが）は、スイートを実行して証拠を手渡すことである。それがコンパニオンキットであり、別売である。ここが、無償の解説と有償のツール群が出会う継ぎ目である。

キットは手法一式をすべて実行する。Hardshell v2 の静的スイープ、PyRIT と garak による敵対的攻撃バッテリー、ログと証拠の検査、そしてアテステーションチェックである。運用ルール（日付付きレポート、不合格の変更キューへのルーティング、シークレットの秘匿化）を自動的に適用し、実行全体を**単一の成果物に統合する。すなわち、すべてのテスト識別子とその合格/不合格を載せた1つの JSON ペイロードであり、CISO が読める PDF にレンダリングされ、RFC 3161 の信頼できるタイムスタンプで封印される。** あなたを信頼しない相手にとって重要なのは、このタイムスタンプの部分である。結果が特定の時点で存在し、それ以降編集されていないことを証明するため、「我々はこれをテストした」は主張から検証可能な記録へと変わる。この単一の署名付き成果物こそ、第1章の最後の問い、すなわち*以上のすべてを、我々を信頼しない相手に証明できるか*に対する具体的な答えであり、エンタープライズの購買担当者や規制当局が実際に見せてほしいと求めてくる納品物である。

検証は、本書の連鎖が終端し、その主張が確認可能になる場所である。残るのは、技術システムを取り囲む人間の説明責任である。誰がリスクを所有するのか、誰がその判断をレビューしたのか、コントロールが破られたとき誰が責任を負うのか。それがガバナンスであり、最終章の主題である。
