# エージェントセキュリティという問題

## モデルが行動できるようになると何が変わるか

テキストを書くだけの言語モデルは、コンテンツ上のリスクである。誤ることも、偏ることも、不快な出力をすることもあり、それらは現実の問題だが、影響範囲（blast radius）は、出力を読んだ人間がその後に何をするかで終わる。人間はまだループの中にいて、ペンを握っている。

*エージェント*は、ある特定の一点において異なる。ループを閉じるのだ。エージェントは自身の出力を受け取り、それをツールに流し込む。API呼び出し、シェルコマンド、データベースへの書き込み、メール送信、支払い。ペンを握っていた人間は今や、よくても流れていくアクションの列を眺めているだけであり、最悪の場合は眠っている。モデルのテキストはもはや提案ではない。それを実行する機械への命令である。

このただ一つの変化こそが、本書の主題のすべてである。下流のすべてはここから導かれる。

- モデルは今や**実効的な権限**を持つ。IAMポリシーに書かれた権限ではない。与えられたツールを通じて実際に到達できる権限を、合成したものである。ブラウザとファイルシステムとメールツールを持つエージェントは、どのツールも単体ではそのために作られていなくても、ファイルから秘密情報を読み取ってメールで送り出すという*合成された*能力（composite）を持つ。
- モデルは、平易な自然言語で**その権限を使うよう言いくるめられうる**。これは従来のソフトウェアに相当物が存在しない性質である。SQLインジェクションには構文上の欠陥が必要だ。エージェントは、Webページに隠された文法的に正しい英語の一文で侵害されうる。モデルは指示に従うように作られており、*あなたの*指示と攻撃者の指示を確実には区別できないからである。
- 帰結は**持続する**。エージェントはメモリに、RAGストアに、ログに、自身の将来のコンテキストに書き込む。攻撃の成功は単発の悪い出力にとどまらない。誰も見ていない3セッション後に発火する、汚染された事実になりうる。

この章から一つだけ考えを持ち帰るなら、これを持ち帰ってほしい。**エージェントとは、見知らぬ他人が話しかけられる特権付き自動化アカウントである。**本書のすべてのコントロールは、「特権付き」の部分か「見知らぬ他人が話しかけられる」の部分のいずれかを狭めるための手段である。

## なぜ既存のセキュリティモデルは必要だが十分ではないのか

アプリケーションセキュリティ、クラウドセキュリティ、アイデンティティエンジニアリングを捨ててよいわけではない。そのすべてが必要である。エージェントはインフラの上で動き、クレデンシャルを保持し、ネットワーク越しのサービスを呼び出し、信頼できない入力を処理する。したがって古典的なコントロールはすべて依然として適用される。秘密情報が平文で誰からも読める状態にあるなら、エージェント固有のどんな工夫もあなたを救わない。

しかし、エージェントの3つの性質が、これらのモデルが立脚してきた前提を壊す。

### 1. 命令とデータの間の信頼境界が消えた

古典的なセキュリティは、制御とデータを分離できるという考えの上に成り立っている。コードはコード、入力は入力であり、仕事は入力がコードとして実行されるのを防ぐことだ。インジェクション攻撃は、その境界が漏れたときに起きる。

言語モデルには、守るべき境界が存在しない。命令もデータも同じチャネル、すなわち自然言語で到着し、同じ機構で処理される。カレンダー招待のメモ、プルリクエストのコメント、PDFの一段落、検索が返した1行。そのすべてがテキストとしてモデルのコンテキストに入り、すべてのテキストは潜在的に命令である。SQLのようにエスケープやパラメータ化で切り抜けることはできない。プロンプトインジェクション（TH-01）が脅威分類の頂点に位置し、他のほぼすべての脅威に関わるのはこのためである。

### 2. 能力は合成的かつ創発的である

古典的なシステムでは、権限はコンポーネント単位で考える。エージェントの実際の能力は、手の届くすべてのツールの*合成*であり、その合成は総和よりも大きい。ツールAはファイルを読む。ツールBはHTTPリクエストを送る。単体ではどちらも危険ではない。組み合わされば、持ち出し（exfiltration）のプリミティブになる。ツールを一つずつ守ることはできない。エージェントがツールを連鎖させて何を達成できるかを考えなければならず、これはより難しく、馴染みの薄い分析である。

### 3. 委譲のもとでアイデンティティと監査が曖昧になる

エージェントは人間の代理として行動し、次にサブエージェントを生成し、それがプロトコル越しに外部のエージェントを呼び、それが共有クレデンシャルを使う。3ホップ目には、「誰が、誰の権限のもとでこれを行ったのか」に答えるのは本当に難しくなり、攻撃者はその曖昧さの中に隠れられる（TH-05, TH-06）。従来のアイデンティティは、リクエストを通じて安定したプリンシパル、すなわち人間かサービスアカウントを前提とする。エージェントシステムはプリンシパルを動的に生成し、信頼ドメインをまたいで権限を委譲する。これはまさに、あなたの監査ログが設計時に想定していなかった状況である。

## 攻撃者の4つの前提

本書のこれ以降は防御側の視点で構成されているが、有能な攻撃者があなたのエージェントをどう捉えるかを冒頭で述べておく価値はある。あらゆる脅威クラスに繰り返し現れる前提が4つある。

1. **境界を混乱させる。** データ、命令、ツール出力、メモリ、承認UIの間の線が曖昧であるほど、攻撃は容易になる。攻撃者は、信頼できないテキストが信頼された意思決定に到達する場所を探し回る。
2. **権限を合成する。** 攻撃者が狙うのは、ブラウザ、ファイル、メール、MCPサーバー、API、人間のアイデンティティを*合わせた*権限であって、モデル単体ではない。
3. **持続する。** 攻撃者は、RAG、メモリ、ログ、設定、ツール定義、あるいはあるケースを評価の対象から外す条件など、ペイロードが現在のターンを越えて生き残る場所ならどこにでも入り込もうとする。
4. **監査を打ち破る。** 作業をサブタスクに分割し、委譲し、ツール呼び出しを分散させて、単一のログでは全体像が語れないようにする。

このうちモデルそのものに関わるのは最初の1つだけであることに注意してほしい。残りの3つは、モデルを取り巻く*システム*に関するものである。これが、「より安全なモデル」ではエージェントセキュリティが解決しない理由であり、本書の大半がプロンプトではなくアーキテクチャ、アイデンティティ、運用を扱う理由でもある。

## 本書が貫くチェーン

本書の構成は飾りではない。それは保証がどのように構築されるかについての主張であり、一方向に流れる。

> **脅威（第2章）→ コントロール（第II部）→ 要件（第9章）→ 検証（第10章）。**

- 対処する**脅威**を名指しせずに**コントロール**を正当化することはできない。脅威のないコントロールはコストであり、コントロールのない脅威は黙って受け入れているリスクである。
- **要件**として書き下していないコントロールを**運用**することはできない。「ツールの権限はレビューしている」は願望である。「REQ-010: ネットワークにアクセスできるすべてのツールは明示的な許可リスト（allowlist）を持たなければならない (SHALL)」は主張である。
- **検証**できない要件を**信頼**することはできない。テストのない SHALL はただの飾りである。第10章は、第9章のすべての SHALL が合格/不合格の判定で終端するために存在する。

本書を読み終えたとき、運用しているどのコントロールについても、それを正当化する脅威まで遡り、それが機能することを証明するテストまで辿り下れるようになっているはずだ。そのチェーンのどこかのリンクが欠けていれば、無駄な労力か、管理されていないリスクのどちらかを見つけたことになる。個々の巧妙な防御ではなく、そのトレーサビリティこそが成果物である。

## 「良い状態」とはどのようなものか

成熟したエージェントデプロイメントは、以下の問いに、主張ではなく証拠で答えられる。

- *このエージェントは実際に何ができるのか？* — 推測ではなく、ツールと権限のインベントリ（第3〜5章）。
- *信頼できないコンテンツが不正な振る舞いを指示したら何が起きるのか？* — ポリシー文書ではなく、実行可能なインジェクションテストスイートによる、テスト済みの答え（第6章、第10章）。
- *誰が、誰の権限のもとで、いつ何をしたのか？* — 委譲のホップをまたいで、ログだけから再構成できる（第4章、第7章）。
- *修正が追いつかない速さで攻撃されたらどうするのか？* — 全員招集のパニックではなく、キルスイッチとSLAを備えた、定義済みのランタイムポスチャ（第8章）。
- *以上のすべてを、我々を信頼していない相手に証明できるか？* — 署名され、タイムスタンプが付与された監査アーティファクト（第IV部）。

これらの問いに答えられるチームと答えられないチームの間の隔たりこそ、本マニュアルが埋めるために書かれた隔たりである。第2章は脅威から始まる。
